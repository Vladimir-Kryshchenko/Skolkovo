# RAG: Полная структура интерфейса «База Сколково»
**Тип:** Навигационная карта сайта для ИИ-чат-бота  
**Версия:** 1.0  
**Дата:** 29.05.2026  
**Назначение:** Подробное описание каждой страницы, каждого блока и каждой надписи системы для точной навигации ИИ-агента при ответах пользователям.

---

## ОБЩАЯ АРХИТЕКТУРА СИСТЕМЫ

Система «База Сколково» — это не один сайт, а **6 отдельных веб-интерфейсов** на разных портах, обслуживающих разные аудитории:

| Интерфейс | Адрес | Для кого | Как попасть |
|---|---|---|---|
| Личный кабинет клиента | :8092/dashboard | Клиенты-резиденты | Ссылка из письма (magic-link) |
| Чат-виджет | :8093/chat | Все пользователи | Кнопка «Консультант» внизу справа |
| Дашборд консультанта | :8094 | Менеджеры | Прямой URL (внутренний) |
| Резидентство-Админ | :8091/clients | Менеджеры резидентства | Прямой URL (внутренний) |
| Главная Админ-панель | :8090 | Администраторы | Прямой URL + логин/пароль |
| MCP-сервер | :8080/mcp | ИИ-агенты и разработчики | API-запросы с ключом |

## ИНТЕРФЕЙС 1: ЛИЧНЫЙ КАБИНЕТ КЛИЕНТА (Портал)
**URL:** :8092  
**Аутентификация:** Magic-link на email (без пароля)  
**Аудитория:** Компании, подающие заявку на резидентство Сколково

### Страница входа `/login`
**Что здесь:** Форма входа для клиентов.  
**Блоки:**
- Логотип «🏛 База Сколково»
- Подзаголовок «Личный кабинет резидента»
- Поле ввода email (подсказка: name@company.ru)
- Кнопка «Отправить ссылку для входа»
- Сообщения об успехе (зелёные) и ошибках (красные)

**Процесс:** Клиент вводит email → система отправляет одноразовую ссылку → клиент переходит по ссылке → автоматически авторизуется.

### Страница дашборда `/dashboard`
**Что здесь:** Обзор текущего статуса резидентства клиента.  
**Навигация (верхнее меню):** 📊 Дашборд | 📋 Чек-листы | ⏰ Дедлайны | 📁 Документы | ✨ Генерация  

**Блоки страницы:**

**Блок «📊 Текущая стадия»** (левая колонка):
- Показывает текущую стадию резидентства (например: «Аккредитация»)
- Прогресс-бар с процентом завершения
- Значение: на каком этапе процесса находится компания

**Блок «📋 Прогресс чек-листов»** (левая колонка):
- Список всех чек-листов клиента
- Для каждого: название + процент выполнения + прогресс-бар
- Пустое состояние: «📋 Чек-листы пока не назначены»

**Блок «⏰ Ближайшие дедлайны»** (правая колонка):
- Таблица дедлайнов с колонками: Дедлайн | Дата | Статус
- Просроченные выделены красным, выполненные — зелёным
- Пустое состояние: «✅ Нет ближайших дедлайнов»

**Блок «📁 Статус документов»** (правая колонка):
- Бейджи статусов документов клиента
- Пустое состояние: «📁 Документы пока не назначены»

### Страница чек-листов `/checklists`
**Что здесь:** Подробный список задач и шагов для прохождения резидентства.  
**Блок «📋 Мои чек-листы»:**
- Карточки для каждого чек-листа (2 в ряд)
- На каждой карточке: название + тип + статус + прогресс-бар + список шагов
- Иконки шагов: ✅ выполнен | 🔄 в процессе | ⬜ не начат
- Пустое состояние: «📋 Чек-листы пока не назначены»

### Страница дедлайнов `/deadlines`
**Что здесь:** Все сроки и дедлайны клиента.  
**Блок «⏰ Ближайшие дедлайны»** (левая колонка):
- Таблица: Название | Дата (ДД.ММ.ГГГГ) | Тип | Статус

**Блок «🔴 Просроченные»** (правая колонка):
- Таблица просроченных задач с красной датой и бейджем «Просрочен»
- Пустое состояние: «✅ Нет просроченных дедлайнов»

### Страница документов `/documents`
**Что здесь:** Документы клиента (договоры, соглашения, поданные файлы).  
**Блок «📁 Мои документы»:**
- Таблица: Документ (название) | Роль документа | Статус | Дата
- Пустое состояние: «📁 Документы пока не назначены»

### Страница генерации `/generate`
**Что здесь:** Автоматическое создание документов из шаблонов (договоры, заявки и пр.).  
**Блок «✨ Генерация документа из шаблона»:**
- Описание функции
- Выпадающий список шаблонов
- Кнопка «Сгенерировать»
- Секция «Доступные шаблоны» — карточки с названием и типом шаблона
- После генерации: кнопка скачивания `/download`

## ИНТЕРФЕЙС 2: ЧАТ-ВИДЖЕТ
**URL:** :8093  
**Аутентификация:** Отсутствует (публичный)  
**Аудитория:** Все пользователи, желающие получить консультацию

### Встроенный виджет (кнопка на любом сайте)
**Что это:** JavaScript-виджет, встраиваемый на любой сайт командой:
```html
<script src="http://[адрес]:8093/chat-widget.js" data-skolkovo-chat='{"base":"http://[адрес]:8093","primaryColor":"#6366f1","welcomeMessage":"Добрый день! Чем могу помочь?"}'></script>
```

**Элементы виджета:**
- Плавающая круглая кнопка (56×56px, правый нижний угол) — открывает/закрывает чат
- Панель чата (380×520px):
  - Шапка «Консультант» с опциональным логотипом
  - Область сообщений (сообщения пользователя справа, бота — слева с markdown)
  - Индикатор «Печатает…» во время ожидания ответа
  - Поле ввода + кнопка отправки
- Настройки: цвет, приветственное сообщение, логотип

### Standalone страница чата `/chat`
**Что это:** Отдельная полноэкранная страница чата.  
**Блоки:**
- Шапка: логотип + «Консультант Сколково»
- Область сообщений с историей
- Индикатор «Печатает…»
- Поле ввода (placeholder «Введите сообщение…») + кнопка «Отправить»

**Как работает чат:**
1. Создаётся сессия (POST /api/session)
2. Сообщение отправляется (POST /api/chat)
3. Чат-сервер передаёт запрос в MCP-сервер (:8080)
4. MCP-инструменты выполняют поиск по RAG-базе
5. Ответ возвращается пользователю в формате markdown

**Блок сводной статистики (5 карточек в ряд):**

- 🟠 **Критично (≤3 дн.)** — дедлайны в ближайшие 3 дня

- **Прогресс** — прогресс-бар + процент
- **Ближайший дедлайн** — название + дата + «через X дн.» или «просрочен на X дн.»
- **Статус** — «Просрочен» / «Критично» / «Внимание» / «В порядке»

## ИНТЕРФЕЙС 6: MCP-СЕРВЕР (для ИИ-агентов)
**URL:** :8080  
**Аутентификация:** API-ключ (заголовок X-API-Key или Bearer)  
**Аудитория:** ИИ-приложения и разработчики

**Эндпоинты:**
- `GET /health` — статус сервиса (JSON)
- `POST /mcp` — MCP-протокол (Streamable HTTP)

### Полный список 19 MCP-инструментов:

**Поиск документов:**
- `search_documents(query, category?, status?)` — семантический поиск по нормативным документам
- `get_document(id)` — получить документ по ID
- `list_active_documents(category?)` — список действующих документов

**Мероприятия и конкурсы:**
- `search_events(query?, date_from?, date_to?, category?)` — поиск мероприятий
- `get_event(id)` — мероприятие по ID
- `search_contests(query?, status?, category?)` — поиск конкурсов и грантов
- `get_contest(id)` — конкурс по ID

**Прочие источники:**
- `search_faq(query, category?)` — поиск по FAQ Сколково
- `search_residents(query?, industry?, status?)` — поиск резидентов
- `get_document_links(document_id, link_type?)` — связи документа

**Резидентство клиентов:**
- `get_client_status(client_id)` — статус, стадия, история переходов
- `get_checklist(client_id)` — чек-листы с прогрессом по шагам
- `get_deadlines(client_id, days_ahead?)` — дедлайны клиента
- `get_client_documents(client_id)` — документы клиента
- `list_clients(tenant_id?, stage?, limit?)` — список клиентов
- `create_client(name, inn, email?, phone?)` — создать клиента
- `update_client_stage(client_id, stage, notes?)` — перевести на новую стадию
- `get_templates(template_type?)` — список шаблонов документов

**Мониторинг изменений:**
- `get_recent_changes(date_from?, date_to?, entity_type?, category?)` — лента изменений (документы, новости, НПА, мероприятия, конкурсы)
- `get_source_health()` — состояние источников данных: актуальность, частота обновлений, ошибки

## СТАДИИ РЕЗИДЕНТСТВА (жизненный цикл клиента)

1. **Подача заявки** — компания подаёт документы
2. **Экспертиза** — эксперты рассматривают заявку
3. **Решение** — принимается решение о резидентстве
4. **Договор** — подписание договора
5. **Резидент** — активный статус резидента
6. **Отчётность** — период сдачи отчётов
7. **Продление** — продление статуса резидента
8. **Выход** — завершение резидентства

**Чек-листы** есть для процедур: Вступление | Отчётность | Продление | Выход

## DOCKER-СЕРВИСЫ (инфраструктура)

Система запускается через Docker Compose и включает:

| Сервис | Назначение | Данные |
|--------|------------|--------|
| **qdrant** | Векторная БД для семантического поиска | Эмбеддинги документов |
| **tei** | Генерация эмбеддингов (multilingual-e5-base) | Модель 768 измерений |
| **postgres** | Основная БД: метаданные, клиенты, AI-конфиг | Всё кроме векторов |
| **skolkovo** | Основной сервис (все серверы в одном бинаре) | Документы, отчёты |
| **nginx** (prod) | Обратный прокси + TLS-сертификаты | — |
| **prometheus** (prod) | Сбор метрик (хранение 30 дней) | Временны́е ряды |
| **grafana** (prod) | Визуализация метрик и алертов | Дашборды |

## КАК ПОЛЬЗОВАТЕЛЬ ВЗАИМОДЕЙСТВУЕТ С СИСТЕМОЙ

### Сценарий 1: Клиент хочет узнать статус резидентства
1. Заходит на Портал (:8092) по magic-link
2. Открывает Дашборд → видит текущую стадию и прогресс
3. Или спрашивает в Чат-виджете — бот вызывает `get_client_status` через MCP

### Сценарий 2: Пользователь ищет информацию о Сколково
1. Открывает Чат-виджет на сайте
2. Задаёт вопрос текстом
3. Бот ищет по RAG через `search_documents`, `search_faq`, `search_events`
4. Получает актуальный ответ с источниками

### Сценарий 3: Менеджер переводит клиента на новую стадию
1. Заходит в Резидентство-Админ (:8091) → Клиенты
2. Находит клиента → открывает Карточку
3. Выбирает новую стадию → нажимает «Перевести»
4. История перехода сохраняется

### Сценарий 4: Администратор добавляет новый документ в базу
1. Заходит в Админ-панель (:8090)
2. Нажимает «📥 Парсинг RSS» — система автоматически загружает новые документы
3. Устанавливает статус «Действует» и категорию
4. Нажимает «🧠 Индексация» — документ попадает в RAG
