## БИЗНЕС-ЦЕПОЧКИ (User Journeys)

### Цепочка 1: Парсинг → Классификация → Индексация → Поиск
1. **Начало:** Админ запускает «📥 Парсинг RSS»
2. **Шаг 1:** RSS-парсер скачивает ~20 документов с dochub.sk.ru
3. **Шаг 2:** Документы сохраняются со статусом «на_проверке»
4. **Шаг 3:** Админ устанавливает статус «Действует» и категорию
5. **Шаг 4:** Админ запускает «🧠 Индексация»
6. **Шаг 5:** RAG-сервис chunk'ит документ, создаёт эмбеддинги через TEI, upsert в Qdrant
7. **Шаг 6:** MCP-инструмент `search_documents` выполняет семантический поиск
8. **Конечный результат:** Пользователь получает релевантные документы через чат-бота

### Цепочка 2: Создание клиента → Назначение чек-листа → Отслеживание прогресса
1. **Начало:** Менеджер создаёт клиента через MCP `create_client`
2. **Шаг 1:** Клиент появляется в таблице `/clients`
3. **Шаг 2:** Менеджер открывает карточку клиента `/clients/{id}`
4. **Шаг 3:** Назначает стадию через «Сменить стадию»
5. **Шаг 4:** Система создаёт дедлайны и чек-листы
6. **Шаг 5:** Консультант видит клиента на дашборде :8094
7. **Конечный результат:** Клиент отслеживается по всей воронке резидентства

### Цепочка 3: Мониторинг изменений → Уведомление консультанту
1. **Начало:** Планировщик запускает парсинг всех источников
2. **Шаг 1:** Новые/обновлённые документы фиксируются в `change_events`
3. **Шаг 2:** `source_health` обновляет статус каждого источника
4. **Шаг 3:** MCP `get_recent_changes` возвращает ленту изменений
5. **Шаг 4:** Telegram-нотификатор отправляет алерт консультанту
6. **Конечный результат:** Консультант знает обо всех изменениях в базе

### Цепочка 4: Валидация документа → Генерация отчёта
1. **Начало:** Вызов MCP `validate_document` с текстом и типом процедуры
2. **Шаг 1:** Валидатор проверяет по чек-листу
3. **Шаг 2:** Возвращает отчёт с проблемами и оценкой
4. **Шаг 3:** При необходимости `draft_document` создаёт черновик
5. **Шаг 4:** `generate_document` рендерит PDF/DOCX из шаблона
6. **Конечный результат:** Готовый документ для клиента

## СТАДИИ РЕЗИДЕНТСТВА (жизненный цикл)
1. **Подача заявки** (application) — компания подаёт документы
2. **Экспертиза** (examination) — эксперты рассматривают заявку
3. **Решение** (decision) — принимается решение
4. **Договор** (contract) — подписание договора
5. **Резидент** (resident) — активный статус
6. **Отчётность** (reporting) — сдача отчётов
7. **Продление** (extension) — продление статуса
8. **Выход** (exit) — завершение резидентства

## ТИПЫ ЧЕК-ЛИСТОВ
- **entry** (Вступление) — 5 шагов
- **reporting** (Отчётность) — 5 шагов
- **extension** (Продление) — 5 шагов
- **exit** (Выход) — 4 шага

## СТАТУСЫ ДОКУМЕНТОВ
- **на_проверке** (pending) — ожидает классификации
- **действует** (active) — актуальный нормативный документ
- **устарел** (outdated) — заменён новым
- **архив** (archived) — исключён из работы
- **отклонён** (rejected) — отклонён администратором
