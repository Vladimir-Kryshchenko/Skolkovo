# Эпик D: Расширение MCP и Multi-tenant

**Версия:** 1.0
**Дата:** 29.05.2026
**Статус:** В работе

## Цель

Расширить MCP-сервер новыми инструментами для работы с клиентами, чек-листами, дедлайнами и документами. Реализовать multi-tenant архитектуру для разделения данных клиентов.

## Задачи

### D1. MCP инструменты для клиентов

- [ ] `get_client_status(client_id)` → стадия, дата обновления, история
- [ ] `get_checklist(client_id, procedure_type)` → чек-лист с прогрессом
- [ ] `get_deadlines(client_id, days_ahead)` → дедлайны на N дней вперёд
- [ ] `get_client_documents(client_id)` → список документов клиента
- [ ] `generate_document(client_id, template_id, data)` → сгенерированный файл

### D2. MCP инструменты для поиска

- [ ] `search_events(query, date_from, date_to)` → мероприятия
- [ ] `search_contests(query, status)` → конкурсы/гранты
- [ ] `search_faq(query, category)` → FAQ
- [ ] `search_residents(query, industry)` → реестр резидентов
- [ ] `ask_consultant(question, client_id?)` → ИИ-ответ с контекстом

### D3. MCP инструменты для валидации

- [ ] `validate_document(document_text, procedure_type, client_id?)` → замечания
- [ ] `get_validation_rules(procedure_type)` → правила валидации

### D4. MCP инструменты для подписок

- [ ] `subscribe_to_changes(client_id, categories)` → подписка
- [ ] `unsubscribe_from_changes(client_id, categories)` → отписка
- [ ] `get_subscriptions(client_id)` → текущие подписки

### D5. Multi-tenant архитектура

- [ ] DB-миграция: добавить `tenant_id` во все клиентские таблицы
- [ ] Модель `Tenant`: id, name, api_key, settings (JSON), created_at
- [ ] Middleware: извлечение tenant_id из API-ключа
- [ ] Фильтрация всех запросов по tenant_id
- [ ] Изоляция данных: клиент tenant A не видит данные tenant B
- [ ] Админка — управление тенантами

### D6. MCP хардненинг

- [ ] Rate-limit per tenant (не только per IP)
- [ ] Quota: лимит запросов в день/месяц на тенант
- [ ] Аудит-лог: кто, что, когда запросил
- [ ] Версионирование MCP API (v1, v2)
- [ ] OpenAPI/Swagger-спецификация MCP инструментов

## Зависимости

- E3 (MCP-сервер)
- B (Управление резидентством)
- A (Расширение источников)
- C (ИИ-агенты)

## Критерии приёмки

- Все MCP-инструменты работают с авторизацией
- Multi-tenant изоляция: данные не пересекаются
- Rate-limit и quota работают на уровне тенанта
- Аудит-лог записывает все запросы
- Swagger-документация доступна
